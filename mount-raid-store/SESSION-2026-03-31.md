# Mount RAID Store — Session Report (2026-03-31)

## Problem

The mount script was found hung with `osascript mount volume` blocking indefinitely since 7:33 AM. The osascript process had been running for hours, holding the lock file, preventing any launchd retry from succeeding. The server was reachable on port 445 the entire time.

A stale `RAID Store-3` directory with `d--x--x--x` permissions was also present in `/Volumes` from a previous failed mount.

## Root Cause Analysis

### Why osascript hung

`osascript mount volume` goes through Finder's NetAuth framework. After a sleep/wake cycle caused the existing mount to go stale, the script force-unmounted it and attempted to remount via osascript. NetAuth was left in a dirty state from the dead session and blocked indefinitely. Even when osascript *does* succeed in mounting, it still hangs and never returns — confirmed by testing during this session.

### Why osascript was being used

The script had been through multiple revisions, each fixing real problems with the previous approach. The committed git history (through `32a6a45`, March 11) used `mount_smbfs`, but uncommitted changes switched to osascript because `mount_smbfs` had its own issues (keychain auth failures, the `-N` implicit auth not working with multiple keychain entries). The osascript approach solved those problems but introduced the hang-forever problem.

### Why mount_smbfs -N fails

The keychain has multiple SMB entries (`Mac-Server.local`, `Mac Server._smb._tcp.local`, `192.168.4.237`, `100.76.199.85`). The `-N` flag tells mount_smbfs to find credentials automatically, but it picks the wrong entry or gets confused by duplicates, resulting in "Authentication error" (exit 77).

### Why mount takes 30 seconds

System log tracing confirmed: macOS's Heimdal library attempts Kerberos authentication against the Local KDC (LKDC) before falling back to NTLMv2. The Kerberos attempt always fails (no KDC configured for the server) but takes ~30 seconds to time out. This happens with both hostname and direct IP. It is not configurable via `nsmb.conf` or any user-accessible setting on macOS Tahoe.

### Why the previous 20-second timeout killed the mount

The old `run_with_timeout` used `perl -e 'alarm shift; exec @ARGV'` which sent SIGALRM to mount_smbfs at 20 seconds — right in the middle of the 30-second Kerberos timeout. The mount was being killed before authentication could complete.

## Changes Made

### mount_raid_store.sh

1. **Replaced `osascript mount volume` with `mount_smbfs`** — direct kernel mount, no GUI/NetAuth dependency, returns reliably.

2. **Explicit keychain credential extraction** — uses `security find-internet-password -s Mac-Server.local -r "smb "` to get username and `-w` to get password, then URL-encodes and embeds in the mount URL. Bypasses the unreliable `-N` implicit keychain lookup entirely.

3. **New `run_with_timeout` implementation** — replaced perl alarm with a background watchdog approach (background the command, sleep in a subshell, kill if exceeded). The perl alarm was incompatible with mount_smbfs's kernel mount operation.

4. **Timeout increased from 20s to 45s** — accommodates the 30-second Kerberos auth timeout with headroom.

5. **Mount point creation via sudo** — `diskutil unmount force` deletes the `/Volumes/RAID Store` directory. Since `/Volumes` is root-owned, the script needs sudo to recreate it. A passwordless sudoers entry was added (see below).

6. **Liveness check simplified** — perl-based stat with 5-second alarm to detect stale mounts after sleep/wake.

### com.oshyan.mount-raidstore.plist

- Added `WatchPaths` for `/Library/Preferences/SystemConfiguration/` files to trigger on network changes.
- `ThrottleInterval` reduced from 30 to 10 seconds.

### /etc/sudoers.d/mount-raidstore (new, on client)

```
oshyan ALL=(root) NOPASSWD: /bin/mkdir -p /Volumes/RAID?Store, /usr/sbin/chown oshyan /Volumes/RAID?Store, /bin/rmdir /Volumes/RAID?Store
```

Allows the launchd agent to recreate the mount point directory without a password prompt.

### ~/Library/Preferences/nsmb.conf (client)

```
[default]
soft=yes
notify_off=yes
```

Unchanged from before. `signing_required=no` was tested but reverted — the server enforces signing regardless (see below).

### Server-side changes attempted (Mac-Server)

Ran `sudo defaults write /Library/Preferences/SystemConfiguration/com.apple.smbd SigningRequired -bool NO` and `RequireSigning -bool NO`, toggled File Sharing off/on. The values wrote successfully but macOS Tahoe (26.3.1) enforces SMB signing at the system level regardless. `smbutil statshares` still shows `SIGNING_REQUIRED=TRUE`. This is not configurable on Tahoe.

## Expected Behavior

### Normal operation (at home)

- launchd runs the script every 60 seconds (or on network change).
- If mount is healthy: script exits instantly (<1 second).
- If mount is stale (after sleep/wake): script detects via liveness check, force-unmounts, remounts in ~30-35 seconds.
- If server is unreachable: script logs once, sets unreachable flag, subsequent runs are silent for 10 minutes.

### Away from home

- Script detects non-home network via gateway IP + MAC check.
- Attempts mount via Tailscale IP (resolved from `tailscale status`, falling back to hardcoded `100.76.199.85`).
- Same 30-second mount time expected (Kerberos timeout is per-mount, not network-dependent).
- If Tailscale is down or server unreachable: logs and retries next cycle.

### Failure recovery

- Timeout prevents any mount attempt from hanging longer than 45 seconds.
- Lock file has a 180-second staleness check — even if the script crashes without cleanup, the next run can proceed after 3 minutes.
- Duplicate/suffixed mounts (`RAID Store-1`, `-2`, etc.) are detected and cleaned up.

### Known tolerances

- **30-second remount time**: unavoidable on macOS Tahoe. Happens after every stale mount detection.
- **~90 second worst-case gap**: 60-second check interval + 30-second mount time before the share is accessible again after a stale event.
- **SMB signing overhead**: enforced by Tahoe, ~20-40% throughput reduction vs unsigned. Not fixable currently.
- **Stale `RAID Store-3` in /Volumes**: stuck `d--x--x--x` directory from a previous dead mount. Harmless (different path from canonical mount). Will disappear on reboot.

## What Could Still Go Wrong

1. **Keychain password change** — if the password for Mac-Server.local changes in the keychain, the script will fail to mount. The error log will show "keychain password lookup failed."

2. **macOS update changes mount_smbfs behavior** — Apple could change the auth flow, remove mount_smbfs, or change /Volumes permissions. Monitor after OS updates.

3. **Tailscale IP changes** — the `LAST_KNOWN_TS_IP` fallback is hardcoded. If Tailscale reassigns the IP and the `tailscale status` lookup fails, the mount will use the stale IP. Update the script if this happens.

4. **Sudoers entry lost** — if `/etc/sudoers.d/mount-raidstore` is removed (e.g., by an OS update), the script can't recreate the mount point after force-unmount. Symptom: "ERROR: cannot create /Volumes/RAID Store" in logs.
