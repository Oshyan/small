# Mount RAID Store — Session Report (2026-04-01)

## Problem

The RAID Store mount was being lost repeatedly during normal use. The user was experiencing the mount disappearing every ~60 seconds, making the share unusable.

## Root Cause

The liveness check added in the 2026-03-31 session was causing an **infinite unmount/remount loop**. Every 60 seconds:

1. launchd runs the script
2. Script finds a healthy mount at `/Volumes/RAID Store`
3. Liveness check runs perl `opendir`/`readdir` on the mount point
4. `opendir` fails with `Operation not permitted` (EPERM)
5. Script concludes mount is stale, force-unmounts it
6. Script remounts successfully
7. 60 seconds later, repeat from step 1

The mount was healthy the entire time. The script was **creating** the problem it was designed to solve.

### Why opendir failed

The `opendir` call on an SMB network volume triggers macOS TCC (Transparency, Consent, and Control) enforcement. Specifically, the `kTCCServiceSystemPolicyNetworkVolumes` permission is required. The launchd agent (`com.oshyan.mount-raidstore`) runs as a plain shell script with no bundle ID and no TCC grants. macOS returns EPERM for any directory-listing operation on network volumes from processes without this permission.

Key distinction: EPERM ("Operation not permitted") is a TCC denial, not a filesystem permission error (which would be EACCES, "Permission denied"). The mount's POSIX permissions (`drwx------` owned by `oshyan`) are fine — the denial comes from macOS's privacy framework operating above the filesystem layer.

### Why this wasn't caught in the previous session

The opendir check was tested **interactively** (from a terminal shell), where the terminal app (Ghostty, Terminal, etc.) has its own TCC grants for network volumes. The check passed every time in interactive testing. The failure only manifests when running under launchd without a terminal's TCC context.

### Why stat passed but opendir didn't

TCC enforcement on macOS is granular by syscall type:
- `stat()` / `lstat()` — returns file metadata. **Not gated by TCC** for network volumes. Works under launchd.
- `opendir()` / `readdir()` — reads directory contents. **Gated by `kTCCServiceSystemPolicyNetworkVolumes`**. Blocked under launchd.
- `statvfs()` (used by `df`) — returns filesystem statistics via kernel VFS layer. **Not gated by TCC**. Works under launchd.

This is why the original stat-only check worked, the enhanced opendir check broke, and the df replacement works.

## Changes Made

### mount_raid_store.sh — `is_mount_alive()` function

Replaced the two-phase liveness check (stat + opendir/readdir) with a two-phase check that avoids TCC:

**Before:**
1. `stat -d` with 8s timeout — catches hung mounts
2. `opendir` + `readdir` with same timeout — verifies SMB session is alive by reading directory contents

**After:**
1. `stat -d` with 8s timeout — catches hung mounts (unchanged)
2. `df` with 8s timeout — verifies SMB session is alive by querying filesystem stats via `statvfs()` syscall

The `df` check serves the same purpose as `opendir` (verifying the SMB session can actually service requests, not just that the mount entry exists) but uses a kernel-level VFS call that doesn't trigger TCC.

### Why df is a valid replacement for opendir

The opendir check was added to catch a specific failure mode: after server reboot or sleep/wake, the mount appears in the mount table and `stat` returns cached data, but the SMB session is dead. Any actual I/O to the server hangs or fails.

`df` catches this same failure mode because `statvfs()` must query the SMB server for filesystem capacity/usage information. If the SMB session is dead:
- With `soft=yes` (set in `nsmb.conf`): `statvfs` returns EIO quickly
- Without soft mount: `statvfs` hangs, caught by the 8-second timeout

## Alternative considered: Full Disk Access

Granting Full Disk Access to `/bin/zsh` (or the script) in System Settings > Privacy & Security would also fix the TCC denial. This was not pursued because:
- Granting FDA to `/bin/zsh` gives every script broad access (too permissive)
- Granting FDA to a shell script directly is unreliable on macOS (may not be recognized)
- Avoiding the TCC-triggering call entirely is cleaner and has no permission prerequisites

This remains a fallback option if `df` proves insufficient in some edge case.

## Outstanding risk: sleep/wake recovery

The `df` check has not yet been tested through a full sleep/wake cycle. The expected behavior:
1. Mac sleeps → SMB session dies
2. Mac wakes → script runs, `stat` may pass (cached), `df` hangs or returns EIO
3. Script detects stale mount, force-unmounts, remounts (~30-35 seconds)

This should work based on how `statvfs` interacts with dead SMB sessions, but will be confirmed after the next overnight sleep cycle (morning of 2026-04-02).

## Diagnostic commands for future debugging

```bash
# Check if the remount loop is happening
tail -20 ~/Library/Logs/mount_raidstore.err

# Healthy pattern: just "Check:" lines, no "liveness:" failures
# Unhealthy pattern: "liveness:" + "stale; forcing unmount" every cycle

# Test liveness check components interactively
perl -e 'alarm 8; print((-d "/Volumes/RAID Store") ? "stat:ok" : "stat:fail"); print "\n"'
df "/Volumes/RAID Store" 2>&1 | head -2
smbutil statshares -a 2>&1 | head -5

# Check TCC status for network volumes
sqlite3 ~/Library/Application\ Support/com.apple.TCC/TCC.db \
  "SELECT client, auth_value FROM access WHERE service='kTCCServiceSystemPolicyNetworkVolumes';"

# Check mount state
mount | grep -i raid
ls -la /Volumes/ | grep -i raid
```
